# @repo/types

Shared Zod schemas and TypeScript types that define the wire contract between `apps/api` and `apps/web`. Every API request/response shape is declared once here as a Zod schema, with its TypeScript type inferred via `z.infer` — so the Fastify route validation and the frontend `ky` client / TanStack Query hooks consume exactly the same types. Also re-exports the enums generated by Prisma (`@repo/database`) so consumers get enum values without importing `@repo/database` directly, and depends on `@repo/constants` for shared validation limits.

## Structure

- One folder per domain feature (e.g. `carbonInventories/`, `organizations/`, `users/`, `reductionProjects/`, `badges/`, `chatbot/`, `dashboard/`, ...), each with an `index.ts` that re-exports the whole feature.
- Within a feature, one subfolder per endpoint/action (e.g. `carbonInventories/createCarbonInventory/`), each containing:
  - `schemas.ts` — Zod request/response schemas.
  - `types.ts` — TypeScript types inferred from those schemas via `z.infer`.
- `baseSchemas/` — Zod schemas for the core domain entities (carbon inventory, organization, user, category, methodology version, file, ...), reused across endpoint schemas via `.pick()` / `.omit()`.
- `common/` — small schemas shared across features: pagination, query params, icon names, considered GEI, GWP source.
- `enums.ts` — re-exports Prisma-generated enums (`SubmissionStatus`, `OrganizationRole`, `InventoryStatus`, ...) as the single import point for enum values.
- `zod.ts` — shared Zod helpers: `IdSchema` (numeric-string ID validation) and `listQueryParam()` (normalizes a comma-separated string or array into a trimmed string array).
- `admin/` — schemas specific to admin-only endpoints, following the same per-feature/per-endpoint layout.

## Usage

```ts
import {
  CreateCarbonInventoryRequestSchema,
  type CreateCarbonInventoryRequest,
} from "@repo/types";
```
