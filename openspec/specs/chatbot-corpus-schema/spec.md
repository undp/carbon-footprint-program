# chatbot-corpus-schema Specification

## Purpose

Defines the dormant migration baseline for the future RAG corpus: the `chatbot_corpus_source`, `chatbot_corpus_chunk`, and `chatbot_corpus_ingest_run` tables (all in the `public` schema with the `chatbot_` prefix), their columns, the source-type/scope/status enums, and their foreign-key actions (CASCADE on chunks, SET NULL on ingest-run source to preserve audit history). These tables ship structurally complete but with no readers or writers, no `embedding` column, and no Postgres extension â€” the `pgvector` extension, the `embedding vector(N)` column, and the HNSW index are deferred to `chatbot-rag-ingest` (V1). The schema is deliberately shaped so that V1 migration lands additively without altering any column introduced here.

## Requirements

### Requirement: Corpus source table tracks versioned ingested documents

The migration SHALL create a `chatbot_corpus_source` table in the `public` schema with the following columns:

- `id` (BIGINT, primary key, generated by default as identity)
- `name` (TEXT, not null)
- `version` (TEXT, not null)
- `source_type` (`chatbot_corpus_source_type` enum, not null)
- `scope` (`chatbot_corpus_source_scope` enum, not null)
- `blob_path` (TEXT, nullable)
- `cite_url` (TEXT, nullable)
- `cite_label` (TEXT, nullable)
- `status` (`chatbot_corpus_source_status` enum, not null)
- `embedding_model` (TEXT, nullable)
- `ingested_at` (TIMESTAMP, default `now()`)
- `activated_at` (TIMESTAMP, nullable)
- `deactivated_at` (TIMESTAMP, nullable)

#### Scenario: Migration creates the table cleanly without pgvector

- **WHEN** the migration runs against an empty Postgres 18 database that does NOT have the `vector` extension installed
- **THEN** the migration SHALL succeed and the `chatbot_corpus_source` table SHALL exist in the `public` schema with all columns listed above

#### Scenario: Table accepts a representative dormant insert

- **WHEN** a manual SQL insert specifies `name = 'GHG Protocol'`, `version = 'v04'`, `source_type = 'PDF'`, `scope = 'GLOBAL'`, `status = 'DRAFT'`
- **THEN** the database SHALL accept the row and assign a non-null `id` and `ingested_at`

### Requirement: Corpus chunk table holds chunked content keyed to source

The migration SHALL create a `chatbot_corpus_chunk` table in the `public` schema with the following columns:

- `id` (BIGINT, primary key, generated by default as identity)
- `source_id` (BIGINT, foreign key to `chatbot_corpus_source.id`, ON DELETE CASCADE)
- `chunk_index` (INTEGER, not null)
- `content` (TEXT, not null)
- `page_number` (INTEGER, nullable)
- `section_title` (TEXT, nullable)
- `tokens` (INTEGER, nullable)

The `embedding` column is intentionally **absent** from this change. It is added in `chatbot-rag-ingest` (V1) cohesively with the `pgvector` extension and the HNSW index, in a single migration.

#### Scenario: Migration creates the table without an embedding column

- **WHEN** the migration runs against an empty Postgres 18 database without the `vector` extension installed
- **THEN** the migration SHALL succeed and the `chatbot_corpus_chunk` table SHALL exist with all columns listed above and SHALL NOT contain an `embedding` column

#### Scenario: Foreign key enforces referential integrity to corpus source

- **WHEN** an insert into `chatbot_corpus_chunk` references a `source_id` that does not exist in `chatbot_corpus_source`
- **THEN** the database SHALL reject the insert with a foreign key violation

#### Scenario: Cascade delete removes chunks on source deletion

- **WHEN** a `chatbot_corpus_source` row is deleted
- **THEN** all `chatbot_corpus_chunk` rows referencing it via `source_id` SHALL also be deleted

#### Scenario: Table accepts a representative dormant insert

- **WHEN** a manual SQL insert specifies `source_id = <existing source id>`, `chunk_index = 0`, `content = 'lorem ipsum'`, with `page_number`, `section_title`, and `tokens` left NULL
- **THEN** the database SHALL accept the row and assign a non-null `id`

### Requirement: Corpus ingest run table audits ingestion attempts

The migration SHALL create a `chatbot_corpus_ingest_run` table in the `public` schema with the following columns:

- `id` (BIGINT, primary key, generated by default as identity)
- `source_id` (BIGINT, foreign key to `chatbot_corpus_source.id`, **nullable**, `ON DELETE SET NULL`)
- `started_at` (TIMESTAMP, default `now()`)
- `completed_at` (TIMESTAMP, nullable)
- `chunks_created` (INTEGER, nullable)
- `embedding_model` (TEXT, nullable)
- `triggered_by` (TEXT, nullable)

The `source_id` column is nullable and the foreign key declares `ON DELETE SET NULL` so that ingest-run rows persist as audit history when a referenced source is deleted. This trade is deliberate: the corpus tables are a versioned audit trail and losing per-source ingestion history on source removal would defeat that purpose.

#### Scenario: Migration creates the table cleanly

- **WHEN** the migration runs against an empty Postgres 18 database
- **THEN** the migration SHALL succeed and the `chatbot_corpus_ingest_run` table SHALL exist with all columns listed above

#### Scenario: Foreign key enforces referential integrity to corpus source

- **WHEN** an insert into `chatbot_corpus_ingest_run` references a non-NULL `source_id` that does not exist in `chatbot_corpus_source`
- **THEN** the database SHALL reject the insert with a foreign key violation

#### Scenario: Source deletion preserves ingest run rows with NULL source_id

- **WHEN** a `chatbot_corpus_source` row is deleted
- **THEN** all `chatbot_corpus_ingest_run` rows referencing it via `source_id` SHALL persist, with `source_id` set to NULL

#### Scenario: Migration declares SET NULL constraint on source_id

- **WHEN** the migration SQL for `chatbot_corpus_ingest_run` is inspected
- **THEN** the foreign key constraint for `source_id` SHALL specify `ON DELETE SET NULL`

#### Scenario: Table accepts a representative dormant insert

- **WHEN** a manual SQL insert specifies `source_id = <existing source id>`, `started_at = now()`, `completed_at = NULL`, `chunks_created = NULL`, `embedding_model = NULL`, `triggered_by = 'manual'`
- **THEN** the database SHALL accept the row and assign a non-null `id`

### Requirement: Corpus enums encode the canonical values

The migration SHALL create three Postgres enums:

- `chatbot_corpus_source_type` with values `PDF`, `MD`, `URL`, `XLSX`
- `chatbot_corpus_source_scope` with values `GLOBAL`, `NATIONAL`
- `chatbot_corpus_source_status` with values `DRAFT`, `ACTIVE`, `OUTDATED`

The Prisma schema SHALL declare these enums and SHALL re-export them via `@repo/database/enums`.

#### Scenario: Each enum has exactly its canonical values

- **WHEN** the migration runs
- **THEN** each enum type above SHALL exist in the database with exactly the values listed and no others

#### Scenario: Enum rejects non-canonical value on source insert

- **WHEN** a row is inserted into `chatbot_corpus_source` with `source_type` set to a value outside `PDF`, `MD`, `URL`, `XLSX`
- **THEN** the database SHALL reject the insert with an enum-violation error

### Requirement: Corpus tables are dormant in this change

This change SHALL NOT introduce any application code that reads from or writes to `chatbot_corpus_source`, `chatbot_corpus_chunk`, or `chatbot_corpus_ingest_run`. The tables exist solely as a structural baseline. Reading and writing arrives in `chatbot-rag-ingest` (V1).

#### Scenario: Grep test enforces no references to corpus tables

- **WHEN** the test `apps/api/test/features/chatbot/lint/noReferencesToCorpusTables.test.ts` runs
- **THEN** it SHALL execute `grep -rE --include='*.ts' '(chatbotCorpusSource|chatbotCorpusChunk|chatbotCorpusIngestRun|chatbot_corpus_source|chatbot_corpus_chunk|chatbot_corpus_ingest_run)' apps/api/src/` from the repo root and SHALL fail if any match is returned. The `--include='*.ts'` flag narrows the scan to TypeScript source (excluding generated artifacts such as `*.js`, `*.d.ts.map`, JSON, and Markdown that may sit under `apps/api/src/`). The path `apps/api/src/` is the production source root â€” test files live separately under `apps/api/test/` and migration SQL lives under `packages/database/`, so neither the schema/migration assertions nor the lint test itself trip the rule. Foundation production code SHALL NOT reference these three tables in any form, including comments: if a future contributor wants to discuss them, that discussion belongs in `chatbot-rag-ingest` (V1), not in foundation source.

#### Scenario: Test suite does not exercise corpus tables

- **WHEN** the test suite runs end-to-end with `LLM_PROVIDER=mock`
- **THEN** no chat or message persistence path SHALL touch the three corpus tables

### Requirement: Migration is forward-compatible with V1 RAG ingestion

The migration SHALL NOT install the `vector` extension or any other Postgres extension. The schema SHALL be shaped so that V1 can land an additive migration that:

- Installs `CREATE EXTENSION IF NOT EXISTS vector`
- Adds the `embedding` column with type `vector(N)` (final dimension chosen in V1)
- Adds an HNSW index on the `embedding` column
- Optionally backfills `embedding_model` on existing `chatbot_corpus_source` rows

â€¦all without altering any column already present in this change.

#### Scenario: Foundation migration contains no extension commands

- **WHEN** the migration SQL generated by this change is inspected (statically â€” by reading the generated `*.sql` file under `packages/database/prisma/migrations/`, or by running the migration against an empty Postgres 18 database without the `vector` extension)
- **THEN** the SQL SHALL NOT contain any `CREATE EXTENSION` statement (case-insensitive match), AND the migration SHALL succeed against a database that has no Postgres extensions installed beyond the Postgres defaults â€” this is the foundation-executable checkpoint that protects the V1 forward-compatibility requirement before V1 lands

#### Scenario: V1 ingest migration applies cleanly on top of foundation

- **WHEN** a future migration adds the `vector` extension, the `embedding` column on `chatbot_corpus_chunk`, and an HNSW index, on top of a database where the foundation migration has already run
- **THEN** the V1 migration SHALL apply without modifying any column shipped in this change (this scenario is intent-only at foundation time â€” V1 verifies it concretely when its migration ships)
