## ADDED Requirements

> Identity resolution, cookie security, and conversation lifecycle (lazy creation, `expires_at` semantics, "active at turn start") are defined in `chatbot-conversation-persistence`. This capability assumes those rules are in force and focuses on the SSE contract, token-cap enforcement, message persistence within an existing conversation, and the error surface for the `POST /api/chatbot/message` endpoint.

### Requirement: Endpoint accepts user message and streams the assistant response via Server-Sent Events

The system SHALL expose `POST /api/chatbot/message`. The request body SHALL include the user's message text. The response SHALL be served as Server-Sent Events with `Content-Type: text/event-stream` and SHALL stream assistant token chunks as `data: ...` events terminated by a final completion event. The response status SHALL be 200 from the moment the response head is sent.

#### Scenario: Successful streaming response

- **WHEN** an authenticated or anonymous client posts a valid chat message
- **THEN** the response SHALL have HTTP status 200, `Content-Type: text/event-stream`, and SHALL emit at least one `data: ...` event followed by a terminal event indicating completion

#### Scenario: Stream events carry assistant content incrementally

- **WHEN** the LLM provider yields multiple token chunks for one assistant turn
- **THEN** each chunk SHALL be flushed as its own SSE `data: ...` event before the next chunk arrives, so a streaming client observes incremental output

### Requirement: SSE response declares the streaming-friendly headers

The response for the streaming endpoint SHALL include the following headers, in addition to `Content-Type: text/event-stream`:

- `Cache-Control: no-cache, no-transform`
- `X-Accel-Buffering: no`
- `Connection: keep-alive`

#### Scenario: All streaming headers present on the response

- **WHEN** the streaming endpoint begins responding to a valid request
- **THEN** the response SHALL carry all four headers above with the values listed

### Requirement: Endpoint accepts both authenticated and anonymous callers

The streaming endpoint SHALL accept both authenticated callers (resolved via `request.currentUser`) and anonymous callers (resolved via the `chatbot_session_id` cookie or a fresh session generated by the identity preHandler). The endpoint SHALL NOT respond with HTTP 401 when no auth is present.

#### Scenario: Anonymous caller served

- **WHEN** an anonymous client posts a valid chat message with no bearer token
- **THEN** the response SHALL be 200 and SHALL stream an assistant response

#### Scenario: Authenticated caller served

- **WHEN** an authenticated client posts a valid chat message with a valid bearer token
- **THEN** the response SHALL be 200 and SHALL stream an assistant response

### Requirement: Endpoint enforces token caps before invoking the LLM provider

The handler SHALL validate the token budget before invoking the `LLMProvider`. The user message token count SHALL NOT exceed `CHATBOT_MAX_USER_INPUT_TOKENS`. The combined token count of the prior conversation history (system + assistant + user messages) SHALL NOT exceed `CHATBOT_MAX_HISTORY_TOKENS`. Token counts SHALL be estimated using the shared `estimateTokens(text)` helper defined in `chatbot-llm-provider`; this handler SHALL NOT inline its own token-count formula. Exceeding either cap SHALL result in HTTP 413 with code `REQUEST_TOO_LARGE` and SHALL NOT invoke the LLM provider.

#### Scenario: Oversized user input rejected

- **WHEN** the request body contains a user message whose `estimateTokens(content)` exceeds `CHATBOT_MAX_USER_INPUT_TOKENS`
- **THEN** the response SHALL be HTTP 413 with body `{ code: "REQUEST_TOO_LARGE", message: ... }` and SHALL NOT include any SSE events

#### Scenario: Oversized history rejected

- **WHEN** the resolved conversation's prior messages add up to an estimated token count exceeding `CHATBOT_MAX_HISTORY_TOKENS`
- **THEN** the response SHALL be HTTP 413 with code `REQUEST_TOO_LARGE` and SHALL NOT invoke the LLM provider

#### Scenario: LLM provider not invoked on cap rejection

- **WHEN** any token-cap rejection occurs
- **THEN** the LLM provider's `streamCompletion` method SHALL NOT be called for that request

### Requirement: Endpoint enforces a per-conversation turn cap

The handler SHALL count user messages in the resolved conversation. A new user message that would bring the count above `CHATBOT_MAX_TURNS_PER_CONVERSATION` SHALL be rejected with HTTP 413 and code `REQUEST_TOO_LARGE`. The LLM provider SHALL NOT be invoked.

#### Scenario: Turn cap rejection

- **WHEN** the conversation already contains `CHATBOT_MAX_TURNS_PER_CONVERSATION` user messages and a new chat message arrives
- **THEN** the response SHALL be HTTP 413 with code `REQUEST_TOO_LARGE` and SHALL NOT invoke the LLM provider

### Requirement: Streamed assistant message is persisted with per-turn metrics on success

The handler SHALL insert a `chatbot_chat_message` row for the assistant turn with `role = ASSISTANT`, an empty `content`, `tokens_used = NULL`, and `latency_ms = NULL` immediately before invoking the LLM provider, capturing the row's `id` for observability. On successful stream completion, the handler SHALL update that row with the accumulated content, `tokens_used` set to the **per-turn total** (`usage.inputTokens + usage.outputTokens` from the provider's terminal `usage` event), `latency_ms` (wall-clock milliseconds from request start to stream end), and `truncated = false`. The assistant row is the canonical home of per-turn cost; user rows always have `tokens_used = NULL`. Summing `tokens_used` across all messages in a conversation therefore yields total cost without double-counting input tokens.

#### Scenario: Assistant row exists before the first chunk

- **WHEN** the LLM provider's `streamCompletion` is about to be invoked for a turn
- **THEN** a `chatbot_chat_message` row SHALL already exist for the assistant turn with `role = ASSISTANT`, `content = ""`, and `latency_ms = NULL`, and SHALL exist BEFORE the first chunk is yielded by the provider

#### Scenario: Successful turn persists final assistant content with per-turn tokens_used

- **WHEN** an LLM stream completes successfully and the provider's terminal `usage` event reports `inputTokens` and `outputTokens`
- **THEN** the corresponding `chatbot_chat_message` row SHALL have non-empty `content`, `tokens_used = inputTokens + outputTokens`, `latency_ms` set to a non-negative integer, and `truncated = false`

### Requirement: Mid-stream client disconnect persists `truncated = true`

If the client disconnects (TCP close, browser navigation away) before the assistant stream completes, the handler SHALL run a `reply.raw.on('close')` finalizer that updates the in-flight `chatbot_chat_message` row with `truncated = true` and the partial accumulated `content`. `tokens_used` and `latency_ms` MAY be NULL or partial. The server SHALL NOT attempt to replay the stream on a subsequent request.

#### Scenario: Disconnect mid-stream marks message truncated

- **WHEN** a streaming client disconnects after at least one token has been emitted but before the terminal completion event
- **THEN** the in-flight `chatbot_chat_message` row SHALL be updated with `truncated = true` and `content` set to the accumulated partial output

#### Scenario: Disconnect before any token marks message truncated with empty content

- **WHEN** a streaming client disconnects before any token has been emitted (provider had not yet yielded its first chunk)
- **THEN** the in-flight `chatbot_chat_message` row SHALL be updated with `truncated = true` and `content` set to the empty string

#### Scenario: Server does not buffer stream for replay

- **WHEN** a client reconnects with `Last-Event-ID` after a mid-stream disconnect
- **THEN** the server SHALL treat the reconnection as a new request, SHALL NOT replay tokens from the prior stream, and the prior assistant message SHALL remain `truncated = true`

### Requirement: User message is persisted before invoking the LLM provider

The handler SHALL insert a `chatbot_chat_message` row for the user turn with `role = USER`, the request's `content`, `tokens_used = NULL`, and `latency_ms = NULL`, before invoking the LLM provider. `tokens_used` is intentionally NULL on user rows because token usage is reported per-turn on the assistant row at finalization (see "Streamed assistant message is persisted with metrics on success"). This guarantees that the user message exists in the conversation regardless of provider success or disconnect.

#### Scenario: User message persists with tokens_used NULL

- **WHEN** the streaming endpoint persists the user message at the start of a turn
- **THEN** the row SHALL have `role = USER`, `content` set to the user input, `tokens_used = NULL`, and `latency_ms = NULL`

#### Scenario: User message persists even when provider fails

- **WHEN** the LLM provider throws before yielding any token
- **THEN** the user message SHALL already be persisted in `chatbot_chat_message` with `role = USER` and `tokens_used = NULL`

### Requirement: Successful turn ends with a `done` SSE event carrying usage

When the LLM provider's stream completes successfully, the handler SHALL emit exactly one terminal SSE event with `event: done` and a `data:` payload carrying the per-turn token usage from the provider. The shape of this event is part of the API contract — the widget hook detects "stream finished" by observing the `done` event, and integration tests assert on it.

The terminal `done` event SHALL have the following exact wire shape:

```
event: done
data: {"inputTokens":N,"outputTokens":M}

```

(That is, an `event: done` field, a `data:` field whose value is the JSON object literal with keys `inputTokens` and `outputTokens` populated from the provider's `usage` event, and the trailing blank line that terminates an SSE event.) `N` and `M` are non-negative integers.

#### Scenario: Successful stream emits the documented done event

- **WHEN** the LLM provider yields its full stream and a terminal `usage` event with `{ inputTokens: 12, outputTokens: 34 }`
- **THEN** the response SHALL emit exactly one terminal SSE event with `event: done` and `data: {"inputTokens":12,"outputTokens":34}` followed by the SSE blank-line terminator, and the connection SHALL close cleanly

#### Scenario: Done event is the last event on the stream

- **WHEN** the response emits any number of `data: ...` chunk events for assistant content
- **THEN** the very last event before connection close on a successful turn SHALL be the `done` event — no further `data:` content events SHALL follow it

### Requirement: LLM provider failures surface as `EXTERNAL_SERVICE_ERROR`

When the configured `LLMProvider` raises an error (timeout, upstream 5xx, abort) before the response head has been sent, the handler SHALL respond with HTTP 503 and code `EXTERNAL_SERVICE_ERROR` (via the `ExternalServiceError` factory). The handler SHALL NOT leak the underlying provider's error message to the client; the message SHALL be a stable, generic phrase. When the failure happens after the response head has been sent (mid-stream), the handler SHALL emit a terminal SSE error event with the exact wire format defined below and rely on the disconnect handler to mark the assistant row `truncated = true`.

The terminal SSE error event SHALL have the following exact wire shape:

```
event: error
data: {"code":"EXTERNAL_SERVICE_ERROR","message":"<stable generic phrase>"}

```

(That is, an `event: error` field, a `data:` field whose value is the JSON object literal with keys `code` and `message`, and the trailing blank line that terminates an SSE event.) The shape of this event is part of the API contract — both the widget hook and the integration tests consume it.

#### Scenario: Provider failure before stream starts

- **WHEN** the configured `LLMProvider.streamCompletion` rejects before yielding the first chunk
- **THEN** the response SHALL be HTTP 503 with body `{ code: "EXTERNAL_SERVICE_ERROR", message: ... }` and SHALL NOT contain provider-specific details

#### Scenario: Provider failure mid-stream emits the documented error event

- **WHEN** the configured `LLMProvider.streamCompletion` errors after at least one chunk has been emitted
- **THEN** the response SHALL emit exactly one terminal SSE event with `event: error` and `data:` set to a JSON object `{"code":"EXTERNAL_SERVICE_ERROR","message":"<stable generic phrase>"}`, the connection SHALL close, and the in-flight assistant row SHALL be marked `truncated = true`

#### Scenario: Mid-stream error event omits provider-specific details

- **WHEN** the underlying provider error contains stack traces, upstream IDs, or vendor-specific fields
- **THEN** the SSE error event's `data.message` SHALL be a stable generic phrase identical to the one used for pre-stream 503 responses, and SHALL NOT include any of those provider-specific details

### Requirement: Route schema declares response shapes for all relevant HTTP status codes

The route schema in `apps/api/src/features/chatbot/sendMessage/route.ts` SHALL declare Zod schemas for every HTTP status code the endpoint can return: 200 (streaming success), 400 (malformed request body), 413 (`REQUEST_TOO_LARGE`), 500 (unexpected server error), and 503 (`EXTERNAL_SERVICE_ERROR`). Schemas SHALL live in `packages/types/src/chatbot/sendMessage/` and be reused.

The 200 response schema documents the SSE event payload shapes (`delta`, `done`, `error`) for OpenAPI/Swagger generation but is NOT applied as Fastify validation or serialization, because the handler hijacks the response via `reply.hijack()` and writes the SSE wire format directly. This means Fastify's Zod type provider does not serialize a JSON body for 200 responses — the schema's role for this status code is purely documentation.

#### Scenario: Schemas exist for each status code

- **WHEN** the route definition is inspected
- **THEN** the `schema.response` map SHALL include keys `200`, `400`, `413`, `500`, and `503`, each pointing at a Zod schema

#### Scenario: Malformed request body rejected with 400

- **WHEN** a request body fails Zod validation (e.g., missing required field)
- **THEN** the response SHALL be HTTP 400 with the standard `ApiErrorResponseSchema` shape and SHALL NOT invoke the LLM provider

### Requirement: Endpoint URL is `POST /api/chatbot/message`

The endpoint SHALL be exposed at exactly `/api/chatbot/message` under the API base path. The route SHALL be registered at `apps/api/src/routes/api/chatbot/index.ts`, following the autoload convention used by the rest of the API.

#### Scenario: Endpoint reachable at canonical URL

- **WHEN** a `POST` is made to `/api/chatbot/message`
- **THEN** the route SHALL be matched and the streaming handler SHALL run
