## Why

Frontend authentication is hard-coupled to Azure via `@azure/msal-browser`/`@azure/msal-react`, so the identity provider cannot be swapped per deployment. The backend already validates tokens from any OIDC issuer that emits the claims it requires (`sub`/`oid` + `email`/`preferred_username`) through the `jwks` provider (proven daily on local + on-prem RD against Entra External ID), which leaves the MSAL→generic-client swap as the only unproven work. This change makes the IdP selectable by environment variable (issuer), unblocking deployments (e.g. MMARN/RD) that cannot or will not use Azure.

## What Changes

- **BREAKING**: Remove `@azure/msal-browser` and `@azure/msal-react` (including their use in `routes/__root.tsx` `MsalProvider` and the `AccountInfo` typing in `hooks/useInitializeUser.ts`); replace with `oidc-client-ts` + `react-oidc-context`. There is no custom login UI — login/logout happen via full-page redirect to the configured IdP. After login the user lands on the `returnTo` carried in the OIDC `state` param, defaulting to `Routes.HOME` when no `returnTo` is set.
- Introduce a singleton `UserManager` (`oidcUserManager.ts`) as the single source of truth for the session **outside React** (used by the `requireRole` route guard and the `ky` HTTP client), plus React hooks for components.
- Add a single public `/auth/callback` route resolved **before** the `/app/*` and `/admin/*` guards. The callback reads `returnTo` from `user.state` and navigates there (else `Routes.HOME`). Session recovery (`handleLoginFailure`, `requireRole` catch) maps the local clear → `userManager.removeUser()`, preserving the existing `authError=login_failed` snackbar + ref-guard behavior.
- The "save draft" flow (`SaveDraftAuthModal`) uses the same full-page redirect, carrying the claim route as `returnTo` in the OIDC `state` so the draft is reclaimed after login — no popup.
- Token storage in `localStorage`, automatic silent renew via the SSO-session-bound refresh token (`offline_access` is an optional realm scope and is NOT requested by the SPA; the `VITE_OIDC_SCOPES` baseline is `openid profile email`), and federated logout (`end_session_endpoint`) — parity with the current MSAL behavior. Greenfield user migration (no `idpUserId` re-mapping).
- Make frontend auth env-driven and build-time wired: `VITE_OIDC_*` keys in `vite-env.d.ts`, `environment.ts`, `apps/web/Dockerfile` (ARG/ENV), **both** `docker-compose.yml` and `docker-compose.prod.yml` (build args), and the `.env*` / `.env.prod.*` templates. Web rebuild is required (Vite is build-time).
- Add a local **Keycloak** dev IdP in docker-compose (split overlays `compose/keycloak-db.yaml` for the shared database + `compose/keycloak.dev.yaml` for the dev IdP) with a reproducible realm export (`infra/keycloak/dev/realm-huella.dev.json`): public `huella-web` client (Auth Code + PKCE S256), audience mapper for `huella-api`, default client scopes `basic profile email access_as_user` with `offline_access` as an **optional** realm scope (NOT requested by the SPA), self-registration on, email required. Critical hostname split: browser `iss` (`http://localhost:18080/...`) vs internal `JWKS_URI` (`http://keycloak:8080/...`).
- Backend env config (`AUTH_PROVIDER=jwks`, `JWKS_ISSUER`, `JWKS_URI`, `JWKS_AUDIENCE`). The API is purely generic JWKS — there is no Azure-specific validation path, and the **auth** `AZURE_*` vars (`AZURE_TENANT_ID`, `AZURE_API_CLIENT_ID`) do nothing in the API runtime. Azure **Storage** `AZURE_*` vars are unaffected. **Scope hardening:** `JwksAuthProvider.ts` reads `payload.scp ?? payload.scope`, `OidcTokenPayload` includes `scope?: string`, and `JWKS_REQUIRED_SCOPE=access_as_user` stays enforced. Keycloak emits `access_as_user` as a default client scope so the check passes.
- Add a **build-time** env-driven Content-Security-Policy in `apps/web/nginx.conf` whose `connect-src`/`form-action`/`frame-src` allow the IdP domain — baked at image build from a build ARG derived from `VITE_OIDC_ISSUER`, no per-country hardcode (the runner has no runtime env, and Vite already forces a rebuild on issuer change, so a runtime `envsubst` model is not used). **Azure SWA is out of scope** for this change: `apps/web/public/staticwebapp.config.json`, `infra/deploy-web.sh`, and `staticWebApp.bicep` are left untouched and stay Azure-only (Option A targets on-prem nginx); making SWA IdP-agnostic is a documented follow-up.

## Capabilities

### New Capabilities

- `generic-oidc-authentication`: Frontend obtains and renews tokens from any OIDC issuer chosen by environment variable, using a singleton `UserManager` shared with the non-React route guard and HTTP client; covers Auth Code + PKCE redirect login, federated logout, automatic silent renew, the public callback ordering, `returnTo`-based post-login navigation, session-recovery mapping, and the redirect-based save-draft flow.
- `oidc-backend-token-validation`: Backend accepts tokens from any env-configured OIDC issuer that emits the required claims via the `jwks` provider (issuer/JWKS-URI/audience contract, browser↔api hostname split, purely generic JWKS validation with no Azure-specific path) plus the generic scope handling (`scp ?? scope` with `JWKS_REQUIRED_SCOPE` enforced).
- `keycloak-dev-environment`: Local Keycloak service + reproducible realm import in docker-compose for development, exposing an issuer consistent between browser and api (hostname split), a PKCE public client, an audience mapper, and self-registration with email required.
- `idp-content-security-policy`: Build-time env-driven CSP in nginx permitting the configured IdP domain (derived from the OIDC issuer via a build ARG) without manual per-deployment edits, scoped to the on-prem nginx deploy path. Azure SWA is explicitly out of scope (Azure-only, untouched, documented follow-up).
