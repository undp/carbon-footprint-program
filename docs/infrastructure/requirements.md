# Infrastructure Requirements

This document describes the recommended compute, storage, and OS specifications for deploying Huella Latam on Azure, along with a summary of cost estimates.

For the full cost breakdown see [`docs/infra cost estimation.pdf`](../infra%20cost%20estimation.pdf).
For the deployment guide see [`docs/Infra/Deployment.md`](../Infra/Deployment.md).

---

## Environment Tiers

The platform is typically deployed in at least two environments:

| Environment | Purpose |
|---|---|
| **Staging** | Pre-production testing, QA, integration validation |
| **Production** | Live system serving real users |

Each environment is an independent Azure Resource Group with its own set of resources.

---

## Azure Services and Sizing

### API — Azure App Service

The API is deployed as a Docker container via Azure App Service.

| Environment | SKU | vCores | RAM | Notes |
|---|---|---|---|---|
| **Development** | B1 (Burstable) | 1 | 1.75 GB | Minimal cost for individual dev environments |
| **Staging** | B2 or P1v3 | 1–2 | 3.5–8 GB | Sufficient for low-to-medium test load |
| **Production** | P2v3 or higher | 2+ | 8 GB+ | Recommended; scale out (multiple instances) as needed |

> The API is **stateless** — horizontal scaling (multiple App Service instances) is supported and recommended for production.

**Minimum requirements:**
- 1 vCore, 1.75 GB RAM for a single-user dev environment
- 2 vCores, 8 GB RAM for staging under realistic load
- Consider auto-scaling rules based on CPU > 70% for production

---

### Frontend — Azure Static Web App

The frontend is a static SPA (pre-compiled HTML/JS/CSS assets). Azure Static Web App serves it via a global CDN.

| Tier | Cost | Notes |
|---|---|---|
| **Free** | $0/month | Sufficient for staging; limited bandwidth |
| **Standard** | ~$9/month | Recommended for production; custom domains, SLA |

No vCore/RAM specification needed — Azure manages CDN infrastructure.

---

### Database — Azure PostgreSQL Flexible Server

**Minimum version: PostgreSQL 15** (the schema uses `NULLS NOT DISTINCT` syntax).

| Environment | SKU | vCores | RAM | Storage | Notes |
|---|---|---|---|---|---|
| **Development** | Standard_B1ms (Burstable) | 1 | 2 GB | 32 GB | Cheapest option (~$12–15/month) |
| **Staging** | Standard_B2s (Burstable) | 2 | 4 GB | 64 GB | Adequate for test data |
| **Production** | Standard_D4s_v3 (General Purpose) | 4 | 16 GB | 128+ GB | Recommended starting point |

**Production notes:**
- Enable **high availability** (zone-redundant standby) for production
- Enable **geo-redundant backups** for disaster recovery
- Configure **backup retention** to at least 14–30 days for production
- Storage autoscaling is available and recommended
- Minimum storage: 32 GB; grows in powers of 2 (32, 64, 128, 256 GB...)

---

### File Storage — Azure Blob Storage

Used for uploaded documents, certificates, and badge images.

| Tier | Notes |
|---|---|
| **Standard_LRS** | Development / staging — locally redundant, lowest cost |
| **Standard_GRS** | Production — geographically redundant |
| **Standard_RAGRS** | Production (critical) — geo-redundant with read access |

**Estimated storage per organization:**
- Small organizations: 10–50 MB (documents + certificates)
- Large deployments (100+ organizations): 1–10 GB total

> All files are private (no anonymous public access). Access is via time-limited SAS URLs generated by the API.

---

### Azure Key Vault

| Tier | Notes |
|---|---|
| **Standard** | Sufficient for all environments |
| **Premium** | Required only if HSM-backed keys are needed |

Low-cost service. Approximately $0.03–0.10/month per secret depending on operations.

---

### Azure Container Registry (ACR)

Used to store Docker images for the API.

| SKU | Notes |
|---|---|
| **Basic** | Development environments — sufficient |
| **Standard** | Staging / Production — recommended |

---

### Azure Front Door (optional)

Used for CDN, WAF, and custom domain with TLS. Recommended for production deployments that require:
- A custom domain (e.g., `app.huellalatam.org`)
- WAF rules (protection against OWASP top 10)
- Global CDN for frontend assets

| SKU | Notes |
|---|---|
| **Standard** | Recommended for production |

## OS and Runtime Requirements

### API Server (App Service)
- **Container OS:** Linux (Alpine or Debian-based Node.js Docker image)
- **Node.js:** ≥ 24.0.0 (as specified in `engines` field of `package.json`)
- **Docker:** Any recent version (used locally and in CI for image builds)

### Database Server
- **OS:** Managed by Azure (Ubuntu-based)
- **PostgreSQL:** ≥ 15 (currently deployed on PostgreSQL 18)

### Local Development
- **OS:** Any (macOS, Linux, Windows with WSL2 recommended)
- **Node.js:** ≥ 24.0.0
- **pnpm:** ≥ 10.23.0
- **Docker:** Any recent version (required for local PostgreSQL via Docker Compose)
- **Azure CLI:** ≥ 2.59.0 (required for deployments)

---

## Networking Considerations

- **PostgreSQL Flexible Server firewall:** Must explicitly allow the App Service IP range (or use VNet integration). For local development, developers add their IP manually in Azure Portal.
- **Blob Storage:** Network ACLs set to `Deny` by default; only Azure services bypass. For development, `Allow` or specific IP exceptions are configured.
- **VNet integration:** Optional but recommended for production to isolate API ↔ DB communication.
- **Front Door:** Routes external traffic; App Service can be locked to only accept traffic from Front Door.

## Scaling Considerations

- **API:** Stateless → supports horizontal scaling via App Service scale-out rules. Set auto-scale trigger at CPU > 70%.
- **Database:** PostgreSQL Flexible Server supports read replicas for read-heavy workloads.
- **Frontend:** Azure Static Web App uses global CDN; no manual scaling needed.
- **Storage:** Azure Blob Storage scales automatically.

For high-traffic scenarios (thousands of concurrent organizations), consider:
- Moving to PostgreSQL GeneralPurpose or MemoryOptimized tier
- Adding a Redis cache layer for frequently accessed reference data (methodology, emission factors)
- Enabling PostgreSQL connection pooling (e.g., PgBouncer)
