# Data Structure Report — Master & Transactional Data, Module Consumption, and User Visibility

**Scope:** entire `carbon-footprint-program` monorepo (Huella Latam platform).
**Purpose:** evaluate reporting coverage and identify integration opportunities.
**Evidence base:** Prisma schema (`packages/database/src/prisma/schema.prisma`), 22 SQL migrations (`packages/database/src/prisma/migrations/`), 28 API feature modules (`apps/api/src/features/`), frontend routes/screens (`apps/web/src/routes/`, `apps/web/src/screens/`), seed pipeline (`packages/database/src/prisma/seeds/`), Python ETL (`load_methodologies/`), infrastructure (Bicep, `infra/`), and project documentation (`docs/`).

> Conventions used in this report: entity names are Prisma model names with the physical table name in parentheses. "Soft delete" means a `status` enum transition (usually `ACTIVE → DELETED`), never a SQL `DELETE`. Statements that are inferences rather than direct code evidence are explicitly marked **[Assumption]** or **[Unknown]**.

---

## 1. Executive Summary

Huella Latam is a country-agnostic carbon footprint measurement, management and recognition platform (Fastify API + React SPA + PostgreSQL/Prisma + Azure Blob Storage + Microsoft Entra ID). Its data model contains **39 physical tables and 4 SQL views**, cleanly separable into:

- **Master/reference data (~24 tables):** country catalogs (sectors, subsectors, organization sizes, job positions, main activities), the methodology taxonomy (methodology versions → categories → subcategories → dimensions → dimension values → emission factors), measurement units, badges, system/country parameters, explanations, reduction-plan initiatives, users and organizations.
- **Transactional data (~15 tables):** carbon inventories and their line/input/factor/result chain, submissions (the workflow engine for accreditation, calculation, verification), reduction projects, files, and append-only audit logs (role changes, access logs).

Key findings:

1. **Reporting is concentrated in three surfaces:** the admin dashboard (KPIs + sector/category charts), client-side Excel/ZIP exports (inventory, reduction project, reduction plan, methodology), and the public transparency portal. There is **no server-side reporting/BI layer** and no scheduled exports; all Excel generation happens in the browser (`docs/development/data-export.md`).
2. **Reporting gaps:** audit data (`UserRoleAudit`, `UserAccessLog`) is only partially surfaced (the `/admin/change-history` screen is an "under construction" placeholder); `CountryParameter` has no API or UI at all; emission-factor usage statistics, submission review SLAs, and multi-year emission trends are not reported anywhere.
3. **Integration posture:** the only live external integrations are Microsoft Entra ID (authentication) and Azure Blob Storage (files via SAS URLs). The public, unauthenticated `GET /api/transparency` endpoint is the platform's only open-data interface; it deliberately exposes recognitions but **redacts emission totals**. Email, AI services, background jobs and APM are documented as planned but not implemented (`docs/integrations.md`).
4. **Material risks found:** auto-provisioned users default to `SUPERADMIN` (`apps/api/src/plugins/app/userResolvePlugin.ts:53`, marked `TODO: remove when finishing the demo`); two orphan foreign-key columns on `carbon_inventory` (`organization_branch_id`, `preselected_nodes_id`) reference tables that do not exist; the inventory's `organization_data` JSONB snapshot duplicates `organization_data` table content; one table breaks naming conventions (`reduction_projects` plural vs. singular elsewhere).

## 2. System Overview

| Layer           | Technology                                                                                                          | Location                                                                              |
| --------------- | ------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------- |
| Database        | PostgreSQL 15+ (Azure Flexible Server), Prisma ORM (preview feature `views`)                                        | `packages/database/`                                                                  |
| API             | Fastify, feature-per-folder (`route.ts → handler.ts → service.ts → helpers.ts`), Zod contracts in `packages/types/` | `apps/api/`                                                                           |

| Auth            | Microsoft Entra ID — pluggable providers: `jwks`, `easy-auth`, `forced-user` (dev), `none`                          | `apps/api/src/auth/`, `docs/security/authentication.md`                               |
| Files           | Azure Blob Storage, Managed Identity, two-step SAS upload flow (15-min URLs)                                        | `apps/api/src/plugins/app/blobStoragePlugin.ts`, `docs/infrastructure/FileStorage.md` |
| Infra           | Azure Bicep modules                                                                                                 | `infra/`                                                                              |
| Master-data ETL | Python scripts converting methodology Excel workbooks → seed JSON                                                   | `load_methodologies/`                                                                 |
| Seeds           | 15 ordered seed scripts, ~242 KB JSON of master data                                                                | `packages/database/src/prisma/seeds/`                                                 |

**Role model (two dimensions):** system roles `USER / ADMIN / SUPERADMIN` and organization-scoped roles `VIEWER / CONTRIBUTOR / ADMIN` (`user_organization_membership.role`). Domain-specific access hooks exist for carbon inventories (including anonymous access via `x-carbon-inventory-uuid` header) and reduction projects (`apps/api/src/plugins/app/*AuthorizationPlugin.ts`).

**Core business flow** (`docs/overview/submission-workflow.md`): an organization registers and requests **accreditation**; members build **carbon inventories** (simplified/expert wizard), **self-declare** them, and request **calculation** and **verification** reviews; verified inventories enable **reduction projects**; each step is a `Submission` reviewed by an admin, and approvals grant **badges** that surface in the user's "Recognitions" screen and the public **transparency** portal.

## 8. Reporting Coverage Analysis

### 8.1 Existing reporting surfaces

1. **Admin dashboard** (`features/dashboard/admin/*`, `screens/AdminDashboard/`): KPIs (organizations total/measuring, emissions total/verified, recognitions delivered/under review), sector bar chart (orgs & emissions, top-N), category pie chart per methodology, submissions status summary, optional year filter. Backed by the four SQL views — the de-facto reporting schema.
2. **Admin requests & organizations screens**: submission pipeline status (`submission_summary_view`) and org accreditation/emissions summary (`organization_summary_view`).
3. **End-user "reports"**: Home emissions summary; inventory ZIP export (summary + detail + factors + methodology + evidence); reduction project Excel; reduction plan Excel. All generated **client-side** with ExcelJS (`docs/development/data-export.md`) — no CSV, no PDF, no scheduled/server-side exports.
4. **Public transparency portal**: accredited organizations × year × recognition seals (no quantitative emissions).

### 8.2 Entities represented in reporting

Carbon inventories (aggregated), organizations, submissions, badges/recognitions, sectors, categories, users (activity KPI), reduction projects (list/export only).

### 8.3 Coverage gaps (operationally important data with no reporting)

| Gap                                                                                                                                                                                   | Evidence                                                                 | Impact                                                                             |
| ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------ | ---------------------------------------------------------------------------------- |
| **Audit reporting absent**: `/admin/change-history` is an UnderConstruction placeholder; `UserAccessLog` only feeds one KPI; creator/updater columns on all tables are never surfaced | `apps/web/src/routes/admin/` placeholder routes                          | Compliance/forensic review requires direct DB access                               |
| **Reduction projects not in admin dashboard**: KPIs cover inventories/recognitions but no aggregate of baseline-vs-project reductions (tCO₂e avoided)                                 | `getDashboardKpis` helpers read inventories/orgs/badges/submissions only | The program cannot report total reductions achieved — arguably its headline metric |
| **No multi-year trend reporting**: dashboard year filter shows one year at a time; no time series for org or country emissions                                                        | dashboard endpoints take a single optional `year`                        | Hard to demonstrate progress over time                                             |
| **Emission factor usage analytics**: `carbon_inventory_line_factor` records which factor was applied, but nothing reports factor usage/coverage                                       | no consumer beyond per-inventory export                                  | Methodology owners can't see which factors matter or are stale                     |
| **Submission SLA/throughput**: `reviewedAt` vs `createdAt` is stored but never aggregated                                                                                             | schema fields unused in KPIs                                             | No visibility of review backlog times                                              |
| **`CountryParameter` and per-country segmentation**: all dashboards implicitly single-country; no country dimension in any report despite the multi-country model                     | views don't group by `country`                                           | Regional (Latam-wide) comparison impossible without new work                       |
| **Files/storage reporting**: no report of stored volume, orphaned blobs, or evidence completeness                                                                                     | —                                                                        | Operations blind spot                                                              |
| **System parameters maintainer missing** (read-only public API exists; UI under construction)                                                                                         | `/admin/parameters` placeholder                                          | Behavior switches require DB access                                                |

### 8.4 Redundant / overlapping reporting

- Organization KPIs appear both in `organizations/admin/getOrganizationKpis` and `dashboard/admin/getDashboardKpis` (overlapping org counts) — consistent today but two code paths to keep aligned.
- The inventory ZIP's `metodologia.xlsx` duplicates the standalone methodology export (`methodologies/getMethodologyExport` vs `carbonInventories/getCarbonInventoryMethodologyExport`) — intentional snapshotting, but two serializers of the same structure.
- Sector ranking exists twice: per-inventory (`getSectorRanking`, computed in-app) and program-wide (dashboard sector chart via SQL view).

### 9.2 High-value entities for new integrations

1. **`organization_summary_view` + sector subtotal views** — already aggregation-ready, ideal backing for a national open-data/statistics API (would need a policy decision on exposing emission totals publicly, currently redacted).
2. **`Submission`** — natural webhook/event source (status transitions) for notifying organizations (email via planned Azure Communication Services) or syncing with national registries.
3. **`MethodologyVersion` export** — the JSON/Excel methodology export could become a versioned, shareable artifact between country deployments (the "common foundation" goal in `.claude/CLAUDE.md`); today loading is one-way via Python scripts.
4. **`OrganizationData.taxId`** — hook for validation against national tax/company registries (currently free text, nullable, unvalidated). **[Assumption]** based on field semantics.
5. **`EmissionFactor` (+ gasDetails)** — candidate for synchronization with public factor databases (e.g., national factor registries); current upkeep is manual Excel → seed.

### 9.3 Missing APIs / synchronization opportunities

- **No admin write API for `SystemParameter` / `CountryParameter`** (read-only and none, respectively).
- **No server-side export API** — all reports are browser-generated; an authenticated CSV/XLSX endpoint would enable BI tools (Power BI/Metabase) without DB credentials. `docs/development/data-export.md` already sketches the server-side streaming approach.
- **No event/outbox mechanism** — any future email/webhook integration will need either polling `submission.updatedAt` or an outbox table.
- **Planned but absent** (from `docs/integrations.md`): Azure OpenAI + AI Search (document analysis/RAG), Azure Functions (background jobs: cleanup, reminders), Azure Communication Services (email), Application Insights (APM).

### 9.4 Risks & dependencies for integrators

- BigInt PKs serialized as strings in API contracts; UUIDs exist only on `user`, `carbon_inventory`, `file` — other entities expose sequential ids (enumeration consideration for public APIs).
- Soft-deleted rows remain in tables; any direct-DB integration must replicate the `status` filters (the SQL views encode the canonical filters — integrate against views, not base tables).
- `organization_summary_view` depends on a `system_parameter` row at query time; missing parameter rows change view semantics.
- Anonymous inventory access via `x-carbon-inventory-uuid` header means inventory UUIDs are bearer credentials — must not leak through integrations or logs.

## 10. Risks and Observations (Architecture)

1. **Critical — default role on auto-provisioning:** every new IdP user is created with `role: SystemRole.SUPERADMIN` (`apps/api/src/plugins/app/userResolvePlugin.ts:53`, comment `TODO: remove when finishing the demo`). In any real deployment this grants full maintainer/admin power to anyone who can sign in.
2. **Orphan columns / missing relations:** `carbon_inventory.organization_branch_id` and `carbon_inventory.preselected_nodes_id` are plain BigInt columns with **no corresponding tables or Prisma relations** anywhere in the schema — dead or future fields. Schema TODOs also note unenforced invariants (only one ACTIVE methodology per country; `methodology_version` status integrity).
3. **Data duplication:** `carbon_inventory.organization_data` JSONB snapshot duplicates `organization_data` fields (intentional point-in-time snapshot for anonymous inventories, but two sources of truth for org attributes per inventory); `user.lastAccessAt` denormalizes `user_access_log` (documented trade-off).
4. **Orphan entity:** `country_parameter` has a full table + audit relations but no API feature, no seed and no UI consumer found. Either remove or implement.
5. **Naming inconsistencies:** `reduction_projects` and `submission_subject_reduction_projects` are plural while every other table is singular; enum `InventoryStatus` carries a schema TODO ("refactor statuses, use submission to infer"); `UsageMode` vs `InputType` overlap (`SIMPLIFIED/EXPERT` duplicated at two levels, documented in a schema comment).
6. **Typing risks:** `reduction_projects.implementation_date` is a `String`, not a date — no DB-level validation, sorting is lexical; `considered_gei` is a free `String[]` (constants exist in `packages/constants/` but the DB doesn't constrain values); `emission_factor.gas_details` and `…line_factor.derivation_details` are unvalidated JSONB.
7. **Hard-delete asymmetry:** `user` is hard-deleted while everything else is soft-deleted, yet dozens of audit FKs (`created_by_id`, …) reference users with nullable FKs — deleting a user nulls/breaks attribution history. `user_role_audit`/`user_access_log` use `onDelete: Restrict`, so deletion fails once a user has logs — inconsistent lifecycle.
8. **Constraint visibility:** many invariants live only in raw SQL partial unique indexes (documented as schema comments) that Prisma cannot track — schema drift risk on future migrations; no DB triggers anywhere (all integrity is application-level + partial indexes).
9. **Scalability concerns:** several ranking/summary endpoints aggregate in application code rather than SQL (`getSectorRanking`, `getSubcategoriesRanking`, parts of dashboard KPIs); `getAllUsers` computes activity in memory over all users; client-side ZIP export materializes everything in browser memory (~100–300 MB practical ceiling per `docs/development/data-export.md`); `organization_summary_view` re-evaluates a correlated `system_parameter` lookup per query.
10. **Single-country reporting bias:** views and dashboards have no country dimension despite the multi-country data model — fine for per-country deployments, a gap if any cross-country aggregation is ever needed.

## 11. Recommendations

**Security / correctness (do first)**

1. Change the auto-provisioning default role to `USER` and seed explicit admins (`userResolvePlugin.ts`).
2. Drop or implement `carbon_inventory.organization_branch_id` / `preselected_nodes_id`; decide the fate of `country_parameter` (implement admin CRUD or remove).
3. Convert `reduction_projects.implementation_date` to a date type; constrain `considered_gei` to the enum in `packages/constants/`.

**Reporting** 4. Add reduction-project aggregates (total tCO₂e avoided, by sector/year) to the admin dashboard — headline program metric currently missing. 5. Build the `/admin/change-history` screen on top of the existing `user_role_audit` + creator/updater columns; consider a generic audit table for catalog changes. 6. Add multi-year trend endpoints (the `carbon_inventory_sector_subtotals_view` already supports `groupBy` year). 7. Implement server-side, streamed CSV/XLSX export endpoints (per `docs/development/data-export.md` roadmap) to unblock BI-tool integration and remove browser memory ceilings.
