# Frontend Architecture

This document describes the structure of `apps/web` — the React + Vite frontend — and the conventions for adding new features.

---

## Directory Structure

```
apps/web/src/
├── api/           # HTTP client + TanStack Query hooks (organized by domain)
├── auth/          # MSAL integration: initialization, token acquisition
├── components/    # Reusable UI components (form fields, layout, status chips)
├── config/        # Environment constants, MSAL config
├── contexts/      # React Context providers (AuthProvider, ExplanationProvider)
├── hooks/         # Custom hooks (e.g., useFuzzySearch)
├── icons/         # Custom SVG icon components
├── interfaces/    # Shared TypeScript types (Routes enum, etc.)
├── routes/        # TanStack Router route files (file-based routing)
├── screens/       # Full-page components (one per route)
├── services/      # Non-API services (excel generation, file download)
├── stores/        # Zustand stores (global client state)
├── theme/         # Material-UI theme (palette, typography)
├── utils/         # Helper functions (requireRole, exporters, formatters)
├── main.tsx       # App entry point
└── routeTree.gen.ts  # Auto-generated by TanStack Router plugin
```

## Routing

TanStack Router uses **file-based routing** with automatic code generation.

```
src/routes/
├── __root.tsx              # Global layout; wraps all routes with providers
├── index.tsx               # /
├── app.tsx                 # /app  (layout + role guard for authenticated users)
└── app/
    ├── organizations.tsx   # /app/organizations
    ├── carbon-inventories.tsx
    └── ...
```

**Root layout (`__root.tsx`)** wraps everything with:
- `ThemeProvider` (MUI theme)
- `MsalProvider` (auth context)
- `QueryClientProvider` (TanStack Query)
- `AuthProvider` (app-level auth state)
- Global error boundary and 404 component

**Route guards** use `beforeLoad`:

```typescript
export const Route = createFileRoute("/app")({
  beforeLoad: async () => {
    await requireRole([SystemRole.USER, SystemRole.ADMIN, SystemRole.SUPERADMIN]);
  },
  component: AppLayout,
});
```

**`routeTree.gen.ts`** is regenerated automatically by `@tanstack/router-plugin` for Vite whenever a route file changes. Never edit it by hand.

## State Management

### Server state — TanStack Query

Every API call goes through a **custom hook** under `src/api/query/` or `src/api/mutation/`. Each domain has:

- `keys.ts` — a QueryKey factory keeping cache keys consistent
- `useXxx.ts` — typed query/mutation hooks

Example pattern:

```typescript
// src/api/query/organizations/keys.ts
export const organizationKeys = {
  all: () => ["organizations"] as const,
  detail: (id: string) => ["organizations", id] as const,
  adminAll: () => ["organizations", "admin"] as const,
};

// src/api/query/organizations/useOrganization.ts
export const useOrganization = (id: string) =>
  useQuery({
    queryKey: organizationKeys.detail(id),
    queryFn: () => apiClient.get(`organizations/${id}`).json<Organization>(),
    staleTime: STALE_TIME_MS,
  });
```

### Client state — Zustand

Global client state lives in `src/stores/`. Currently minimal — only the `userStore` for the authenticated user.

```typescript
create<UserState>((set) => ({
  user: null,
  isLoading: false,
  setUser: (user) => set({ user, error: null }),
  clear: () => set({ user: null, isLoading: false, error: null }),
}));
```

Component-local state stays in React (`useState`, `useReducer`). Avoid putting derived server data in Zustand — that's TanStack Query's responsibility.

## HTTP Client

`ky` is configured once in `src/api/http/client.ts`:

```typescript
const apiClient = ky.create({
  prefixUrl: import.meta.env.VITE_API_BASE_URL,
  timeout: 30_000,
  hooks: {
    beforeRequest: [
      async (request) => {
        const token = await getAuthToken();
        request.headers.set("Authorization", `Bearer ${token}`);
      },
    ],
    afterResponse: [normalizeErrorHook], // throws AppHttpError
  },
});
```

Never call `fetch` directly — always go through `apiClient` so authentication and error normalization are applied.

## Authentication

MSAL handles the OAuth2/OIDC flow against Azure Entra ID.

1. `initializeMsal()` is called once during app startup (in `main.tsx` or `__root.tsx`).
2. `msalInstance.handleRedirectPromise()` completes the login redirect.
3. `getAuthToken()` acquires access tokens silently (falls back to popup on consent failures).
4. The `ky` `beforeRequest` hook attaches `Authorization: Bearer <token>` to every API call.

See [MSAL / Easy Auth Setup](../MSAL-EasyAuth-Setup.md) for tenant and app registration configuration.

## Forms

All forms use `react-hook-form` + Zod schema validation:

```typescript
// schema.ts
const schema = z.object({
  name: z.string().min(1, "El nombre es requerido"),
  email: z.string().email(),
});
type FormValues = z.infer<typeof schema>;

// Component
const form = useForm<FormValues>({
  resolver: zodResolver(schema),
  defaultValues: { name: "", email: "" },
});
```

Custom field components in `src/components/` (e.g., `FormAutocompleteField`, `FormTextField`) wrap MUI inputs with `<Controller>` so validation messages flow automatically into the `helperText` prop.

For cross-field validation, use Zod's `.superRefine()`.

## Theme and Styling

**Material-UI theme** (`src/theme/`) defines the color palette, typography, and component overrides. All MUI components pick up these tokens automatically.

**Tailwind CSS** (`tailwind.config.js`) provides utility classes for spacing, layout, and one-off styling. Both can be combined — Tailwind utilities are applied via `className`, MUI styling via the `sx` prop or `styled()`.

**Rule of thumb:**
- MUI components for interactive widgets (buttons, inputs, menus, tables).
- Tailwind for layout primitives (flex, grid, spacing, responsive breakpoints).
- Emotion / `styled()` only when MUI and Tailwind don't cover the case.

## Screens and Components

- **Screens** (`src/screens/`) are route-level components. Each screen is responsible for its own data fetching (via TanStack Query hooks), form state, and layout.
- **Components** (`src/components/`) are reusable UI pieces shared across screens — never make HTTP calls or read from Zustand directly. Pass data in via props.

## Adding a New Screen

Typical workflow for a new feature with a new page:

1. **Add the route file** at `src/routes/app/my-feature.tsx`:
   ```typescript
   export const Route = createFileRoute("/app/my-feature")({
     beforeLoad: async () => {
       await requireRole([SystemRole.USER]);
     },
     component: MyFeatureScreen,
   });
   ```

2. **Create the screen** at `src/screens/MyFeature/MyFeatureScreen.tsx`.

3. **Add the query hook** at `src/api/query/myFeature/useMyFeature.ts` with a matching `keys.ts`.

4. **Define the form schema** at `src/screens/MyFeature/formSchema.ts` (if the screen has a form). The schema can reuse Zod schemas from `@repo/types` when the shape matches the API.

5. **Add mutation hook** at `src/api/mutation/myFeature/useCreateMyFeature.ts` (if the screen submits data).

6. **Update the route path enum** in `src/interfaces/routes.ts` if you add a new named route.

7. **Run the dev server** — TanStack Router regenerates `routeTree.gen.ts` automatically.
