# Data Structure Report — Master & Transactional Data, Module Consumption, and User Visibility

> **Purpose**: complete inventory of the platform's data structures (master and transactional), a mapping of which modules consume each table, and the information ultimately displayed to end users. Intended to evaluate reporting coverage and identify integration opportunities.
>
> **Method & evidence base**: static analysis of this repository only — Prisma schema (`packages/database/src/prisma/schema.prisma`), SQL migrations (`packages/database/src/prisma/migrations/`), seed scripts (`packages/database/src/prisma/seeds/`), API features (`apps/api/src/features/`), API plugins (`apps/api/src/plugins/`), frontend routes/screens (`apps/web/src/routes/`, `apps/web/src/screens/`), infrastructure code (`infra/`), and project documentation (`docs/`). No live database was inspected, so **data volumes are unknown** throughout this report. Assumptions and unknowns are explicitly marked.
>
> **Snapshot date**: 2026-06-10.

---

## 1. Executive Summary

Huella Latam is a country-agnostic carbon footprint measurement, management, and recognition platform (digital public good for Latin America), deployed independently per country. Its data model is a single PostgreSQL schema managed by Prisma:

- **42 tables** (26 classified here as master/reference data, 16 as transactional/operational data) plus **4 SQL views** used as the reporting backbone.
- **28 API feature domains** (`apps/api/src/features/`) follow a uniform `route.ts → handler.ts → service.ts` pattern; all database access goes through Prisma (raw SQL only in the health check, `apps/api/src/routes/health.ts`).
- **The core value chain** is: country catalogs + versioned methodology (categories → subcategories → emission factors) → carbon inventories (line → input → applied factor → result, fully reproducible) → submission/review workflow → recognition (badges) → public transparency.
- **User-facing surfaces**: an anonymous calculator flow, an authenticated organization workspace (inventories, reduction plan/projects, recognitions, organization profile), an admin back office (dashboard, request review, organization & user management), a superadmin methodology maintainer suite, and a public transparency page.
- **Reporting today** is: 1 admin dashboard (6 KPIs + 2 charts), 1 public transparency endpoint, and 4 client-side Excel exports (carbon inventory ZIP, methodology, reduction plan, reduction project). All reporting is synchronous and view-backed; there is **no data warehouse, no scheduled jobs, and no outbound integrations** in application code.
- **Key gaps found**: `user_access_log` is write-only (collected, never reported); `country_parameter` is a dormant table (no API, no seed, no consumer); two `carbon_inventory` columns (`organization_branch_id`, `preselected_nodes_id`) reference entities that don't exist as tables; user deletion is a hard delete that conflicts with the platform's historical-integrity design; email, telemetry, background jobs and AI services are provisioned in infrastructure but not wired into the application.
- **Integration posture**: the platform already exposes a machine-readable OpenAPI contract (Swagger at `/api/docs`), a public transparency endpoint, and an Excel→JSON→seed methodology pipeline — natural starting points for national-registry integrations, BI extraction, and methodology exchange between country deployments.

## 2. System Overview

### 2.1 Architecture

| Layer                  | Technology                                                                      | Location                                     |
| ---------------------- | ------------------------------------------------------------------------------- | -------------------------------------------- |
| Database               | PostgreSQL 15+ (requires `NULLS NOT DISTINCT`)                                  | provisioned per-country (Azure reference)    |
| ORM / schema           | Prisma (`prisma-client`, ESM, `views` preview)                                  | `packages/database/src/prisma/schema.prisma` |
| API                    | Fastify + Zod (`fastify-type-provider-zod`), OpenAPI at `/api/docs`             | `apps/api/`                                  |
| Web                    | React, TanStack Router (file-based), TanStack Query v5, MUI v7 + Tailwind, MSAL | `apps/web/`                                  |
| Shared contracts       | Zod schemas + inferred types                                                    | `packages/types/`                            |
| Shared constants/utils | `@repo/constants`, `@repo/utils`                                                | `packages/constants/`, `packages/utils/`     |
| File storage           | Azure Blob Storage (SAS URLs; Azurite locally)                                  | `apps/api/src/services/blobService.ts`       |
| AuthN                  | Azure Entra ID / Easy Auth / generic JWKS / forced-user (dev)                   | `apps/api/src/auth/providers/`               |
| Infra as code          | Azure Bicep                                                                     | `infra/`                                     |
| Methodology tooling    | Python Excel→JSON pipeline                                                      | `load_methodologies/`                        |

### 2.2 Module model

Each API feature domain owns its endpoints and services. Cross-cutting access control is implemented as Fastify decorators (`requireAuth`, `requireRoles`, `requireOrganizationRole`, `requireCarbonInventoryAccess`, `requireReductionProjectAccess`) in `apps/api/src/plugins/app/*AuthorizationPlugin.ts`. The web app mirrors the domains in `apps/web/src/api/query/<domain>/`.

The 28 API feature domains: `badges`, `carbonInventories`, `categories`, `countryOrganizationSizes`, `countrySectors`, `countrySubsectors`, `dashboard`, `emissionFactorDimensions`, `emissionFactors`, `explanations`, `files`, `forms`, `jobPositions`, `magnitudes`, `measurementUnits`, `methodologies`, `organizationMainActivities`, `organizations`, `reductionPlanInitiatives`, `reductionProjects`, `requests`, `subcategories`, `subcategoryRecommendations`, `submissions`, `systemParameters`, `termsConditions`, `transparency`, `users`.

### 2.3 Role model (two dimensions)

- **System roles** (`SystemRole`): `USER`, `ADMIN`, `SUPERADMIN`.
- **Organization roles** (`OrganizationRole`): `VIEWER`, `CONTRIBUTOR`, `ADMIN` — via `user_organization_membership`.
- **Anonymous access**: the calculator flow allows unauthenticated use of a carbon inventory identified by `x-carbon-inventory-uuid` header (UUID held in browser localStorage), later "claimable" by a logged-in user (`carbonInventories/claimCarbonInventory`).

### 2.4 Data lifecycle conventions

- **Soft deletes via status enums** nearly everywhere (`ACTIVE`/`DELETED`/`OUTDATED`); uniqueness enforced with **partial unique indexes written in raw SQL migrations** (documented as comments in the schema, not tracked by Prisma diffs).
- **Versioning over updates**: `organization_data` rows are versioned (ACTIVE/OUTDATED); `carbon_inventory_line_input` keeps one active row per line (DB constraint `UNIQUE(line_id) WHERE is_active`, migration `20260101235319_*`), replaced inputs are kept.
- **Audit columns** `created_by_id` / `updated_by_id` on most business tables (exceptions noted in §10).

## 5. Entity Relationship Overview

```
            │   │
            │   └──< UserOrganizationMembership >── User ──< UserRoleAudit / UserAccessLog
            │
            ├──< CarbonInventory >── MethodologyVersion
            │        └──< CarbonInventoryLine >── Subcategory
            │                 ├──< CarbonInventoryLineInput >── (dimension values, unit, rate unit)
            │                 │        ├── 1:1 CarbonInventoryLineFactor >── EmissionFactor?
            │                 │        └── 1:1 CarbonInventoryLineResult
            │                 └──< CarbonInventoryLineFile >── File
            │

SubmissionSubject ── 1:1 ── {SubjectCarbonInventory | SubjectOrganizationData | SubjectReductionProject}

                  └──< SubmissionFile >── File
```

Workflow semantics (documented in `docs/data-model/conceptual-guide.md`, `docs/overview/submission-workflow.md`):

- **Accreditation**: `OrganizationData` (ACTIVE draft) → `Submission(ORGANIZATION_ACCREDITATION, PENDING)` → APPROVED (org accredited) / REJECTED (data → OUTDATED, resubmit with new version).
- **Calculation / verification**: inventory → `requestCalculation` (auto-approvable → `APPROVED_AUTOMATICALLY` + badge) → `requestVerification` (third-party verified) → badges surface in Recognitions and Transparency. Self-declaration is an alternative flag on the inventory.
- **Reduction project verification** mirrors inventory verification via its own subject table.
- `NEUTRALIZATION_PLAN_VERIFICATION` exists in enums/badges, but **no neutralization-plan entity or feature module exists yet** (`docs/overview/neutralization-plan.md` describes intent — gap noted in §8/§10).

## 8. Reporting Coverage Analysis

### 8.1 Covered today

| Business question                                         | Surface                                           | Backing data                                             |
| --------------------------------------------------------- | ------------------------------------------------- | -------------------------------------------------------- |
| How many organizations participate / measure?             | Admin dashboard KPIs                              | `organization`, `organization_summary_view`              |
| Total vs verified national emissions (year-filtered)      | Admin dashboard KPIs                              | `carbon_inventory_subtotals_view` + submission status    |
| Emissions by sector / by category                         | Admin dashboard charts                            | `carbon_inventory_sector_subtotals_view`, subtotals view |
| Review pipeline state (pending/approved/reviewed by type) | Admin dashboard + requests KPIs                   | `submission_summary_view`, `submission`                  |
| Who is accredited / recognized (public)                   | Transparency page                                 | `organization_summary_view` + nested submissions         |
| One organization's footprint detail                       | Calculator summary screens + inventory ZIP export | full measurement chain (T1–T6)                           |
| Methodology content                                       | Methodology Excel export                          | M16–M23                                                  |
| Reduction commitments                                     | Project Excel export, projects list               | `reduction_projects`                                     |
| Role-change audit per user                                | Admin users → role history                        | `user_role_audit`                                        |

### 8.2 Coverage gaps (operationally important, not reported)

1. **User activity / adoption**: `user_access_log` and `User.lastAccessAt` are captured but never read — no active-user, adoption, or engagement reporting (evidence: zero read call sites; `docs/development/user-activity-tracking.md` documents collection only).
2. **Emission factor usage**: nothing reports which factors/subcategories are most used or which factors are unused — valuable for methodology curation (data exists in `carbon_inventory_line_factor.emission_factor_id`).
3. **Data-quality / completeness funnel**: the subtotals view computes `active_lines_count` vs `active_completed_lines_count`, but no surface reports completion rates across organizations (drop-off analytics for the calculator funnel are absent).
4. **Year-over-year trends**: every KPI is single-year (`year` filter); no trend/time-series view for emissions or participation.
5. **Reviewer throughput / SLA**: `submission.reviewedAt - createdAt` is computable but unreported (review latency, per-reviewer workload).
6. **Files/storage**: no reporting on stored evidence (counts, volume `sizeBytes`, orphaned blobs).
7. **Input history**: versioned `carbon_inventory_line_input` history (who changed what, when) is retained but not surfaced anywhere, despite being a stated design goal (auditability).
8. **Neutralization plans**: badge/submission types exist (`NEUTRALIZATION_PLAN_VERIFICATION`) but there is no entity, module, or report — the workflow cannot complete end-to-end.
9. **Reduction plan engagement**: `reduction_plan_initiative` is exported but adoption (which orgs downloaded/used which initiatives) is untracked.
10. **Country/system parameters**: `/admin/parameters` and `/admin/change-history` screens are placeholders (`UnderConstruction`).

### 8.3 Redundant or overlapping reporting

- **Organization KPIs computed twice**: `dashboard/admin/getDashboardKpis` and `organizations/admin/getOrganizationKpis` independently aggregate organization counts (different services, same underlying view) — consistency depends on parallel maintenance.
- **Submission KPIs split** between `dashboard` (groupBy on `submission`) and `requests/admin/getRequestsKpis` (groupBy on `submission_summary_view`).
- **Methodology export duplicated by design**: `GET /methodologies/:id/export` and `GET /carbon-inventories/:id/methodology-export` are deliberately byte-identical (the latter allows anonymous access) — acceptable, but contract drift must be guarded by tests.
- All Excel generation is **client-side**; the same "report" logic exists once per export utility with overlapping formatting code (`apps/web/src/utils/export*.ts`, `apps/web/src/services/excel.ts`).

- Partial unique indexes & views live in raw SQL migrations — schema introspection via Prisma alone misses them.

## 10. Risks and Observations (architecture)

1. **Dormant table**: `country_parameter` has no API, no seed, no consumer (repo-wide search). Either implement country-scoped config or remove it.
2. **Dangling references**: `carbon_inventory.organization_branch_id` and `preselected_nodes_id` are plain BigInt columns with **no FK and no target table**, yet they flow through API schemas (`packages/types/src/baseSchemas/carbonInventory.ts`, `updateCarbonInventory`, `duplicateCarbonInventory`). Future "branches"/"preselection" features are implied but unmodeled — integrity is unenforced.
3. **Hard user deletion** (`users/deleteUser` → `prisma.user.delete`) contradicts the historical-integrity principle; with `Restrict` FKs on audit/log tables and dozens of `created_by` references, deletion will fail for any active user (FK violation) and succeed only for pristine accounts — inconsistent semantics vs the soft-delete convention used everywhere else.
4. **Write-only logs**: `user_access_log` grows unbounded (indexed but never read, no retention job — background processing is absent altogether).
5. **Naming inconsistencies**: table `reduction_projects` (plural) vs singular naming everywhere else; likewise `submission_subject_reduction_projects`. `ReductionProject.implementationDate` is a `String` while all other dates are `DateTime`.
6. **Inconsistent audit columns**: `MeasurementUnit`, `RateMeasurementUnit`, `Magnitude`, `Badge`, `File` (partially), and junction tables lack `updated_by`/`created_by` coverage that other catalogs have.
7. **JSONB escape hatches**: `EmissionFactor.gasDetails`, `CarbonInventory.organizationData`, `LineFactor.derivationDetails`, `LineResult.resultDetails` are schema-less; validation exists only in Zod at the API boundary — drift risk for downstream consumers.
8. **Duplicated org data**: anonymous inventories keep an `organizationData` JSONB snapshot that is not reconciled with relational `organization_data` after claiming — two sources of truth for organization attributes on an inventory (assumption: reconciliation not found in `claimCarbonInventory`/`assignOrganizationToCarbonInventory` services; flagged for verification with product owners).
9. **Doc/schema drift**: `docs/data-model/conceptual-guide.md` references a `status_catalog` table that does not exist; seeds docs vs actual scripts diverge slightly. Maintain docs against `schema.prisma`.
10. **Workflow status TODO acknowledged in schema**: `InventoryStatus` carries a TODO ("use submission to infer statuses") — inventory state is partially duplicated between `carbon_inventory` flags (`isEditable`, `isSelfDeclared`) and submission statuses.
11. **Scalability**: reporting relies on non-materialized views with CTE pipelines (`organization_summary_view`) executed per request; transparency loads all accredited organizations with nested includes and post-processes with lodash in app memory (`features/transparency/.../service.ts`) — fine at current scale, a candidate for pagination/materialization as data grows (`docs/operations/scaling-playbook.md` exists but views are recomputed per query).
12. **Enum/feature mismatch**: `NEUTRALIZATION_PLAN_VERIFICATION` exists in `SubmissionType`/`BadgeType` with no implementing module — submissions of that type cannot be created today.
13. **Single methodology activation weakly enforced**: schema TODOs admit there is no DB-level guarantee of "exactly one PUBLISHED methodology per country".

**Unknowns (and what would resolve them):**

- **Data volumes / growth rates** — requires production DB metrics (no DB access from this analysis).
- **Actual per-country configuration in the field** (which auth provider, which seeds customized) — requires deployment inventories; only reference IaC and `.env` templates exist here.
- **`/capinaut` route purpose** — renders `UnderConstruction`; no other references found.
- **Formal data ownership** — no ownership registry exists; owners in this report are inferred from RBAC write paths.

## 11. Recommendations

Priority-ordered; all are evidence-linked to §8–§10.

1. **Close the activity-reporting gap** (low effort, high value): add admin endpoints over `user_access_log`/`lastAccessAt` (active users, adoption funnel) and a retention policy for the log.
2. **Decide `country_parameter`'s fate**: implement the country-config API it implies (needed for true multi-country deployments) or drop the table to reduce schema noise.
3. **Model or remove dangling fields**: create `organization_branch` / preselection entities with FKs, or strip `organizationBranchId`/`preselectedNodesId` from schema and contracts.
4. **Replace hard user delete** with the platform's own soft-delete/anonymization pattern (status + PII scrubbing), consistent with audit FKs.
5. **Unify KPI computation**: extract shared KPI services (organizations, submissions) consumed by both `dashboard` and domain modules to prevent metric drift.
6. **Add trend reporting**: year-over-year emissions/participation series from existing views (no new collection needed).
7. **Introduce eventing for the submission workflow** (webhook or wired Azure Communication Services email) — infra already provisioned; biggest UX/integration win per effort.
8. **Publish an integration-grade read API**: paginated transparency + factor catalog export endpoints with versioned contracts (OpenAPI already in place), enabling registries/BI without DB coupling.
9. **Implement neutralization plans or remove the enum values** to restore enum/feature parity.
10. **Plan view materialization/pagination** for `organization_summary_view` and transparency before national-scale rollout; wire Application Insights to observe real query cost.
11. **Fix naming/typing debt opportunistically** (plural table names, `implementationDate: String`) — schema migrations are cheap now, expensive after federation.
12. **Keep docs synchronized**: correct `status_catalog` reference; document the JSONB contracts (`gasDetails`, `derivationDetails`) in `docs/data-model/`.
